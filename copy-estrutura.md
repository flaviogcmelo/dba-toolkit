# Padrões de Copy e Estrutura de Posts — Flávio

Este documento reúne meus padrões de copywriting para posts técnicos.  
Ele serve como guia prático para criar aberturas fortes, desenvolver ideias com clareza e encerrar com impacto.

---

# 1. Princípios de copy que guiam meus posts

### **1.1 Clareza acima de tudo**
- Uma ideia por parágrafo.
- Frases curtas.
- Evitar floreios desnecessários.
- Priorizar entendimento imediato.

### **1.2 Leveza e naturalidade**
- Escrever como se estivesse conversando com alguém no café.
- Tom humano, acessível e profissional.
- Sem emojis (por enquanto).

### **1.3 Foco no leitor**
- Mostrar por que aquilo importa.
- Conectar o problema ao dia a dia do profissional de TI.
- Antecipar dúvidas e responder no próprio texto.

### **1.4 Valor imediato**
- Entregar algo útil já nos primeiros parágrafos.
- Evitar enrolação.
- Sempre incluir um insight, prática, exemplo ou aprendizado.

# 2. Estrutura base de um post técnico

## **2.1 Abertura (gancho)**
A função da abertura é fazer o leitor parar de rolar o feed.

Ganchos que combinam comigo:
- Pergunta provocativa  
- Mini-história  
- Erro comum  
- Curiosidade técnica  
- Cenário imaginado  

**Exemplos de ganchos no meu estilo:**
- “Por que seu banco fica lento mesmo com índices bem configurados?”  
- “Semana passada, um cliente me chamou desesperado por causa de uma replicação travada.”  
- “Se você ainda usa `SELECT *`, este post é para você.”  
- “Sabia que um único parâmetro mal configurado pode derrubar sua replicação?”  

---

## **2.2 Desenvolvimento**
Aqui é onde entrego o valor.

Minha estrutura preferida:
1. Problema raiz  
2. Causa real  
3. Explicação com analogia  
4. Exemplo prático  
5. Código (se houver)  
6. Insight final  

**Regras do desenvolvimento:**
- Parágrafos curtos.  
- Linguagem simples, mas técnica.  
- Explicar termos na primeira vez.  
- Mostrar o raciocínio, não só a solução.  

---

## **2.3 Encerramento**
O final deve:
- reforçar o aprendizado,  
- trazer uma conclusão pessoal,  
- convidar à interação.

**Modelo de encerramento no meu estilo:**
> “Em resumo, o problema não estava no índice, mas na forma como a consulta era construída.  
> Se você já passou por algo parecido, compartilha aí — sempre dá para aprender mais.”

# 3. Frameworks de copy para facilitar a escrita

## **3.1 Framework “Problema → Causa → Solução”**
Ideal para posts técnicos diretos.

**Estrutura:**
- O que parece ser o problema  
- O que realmente é o problema  
- Como identificar  
- Como resolver  
- Como evitar  

---

## **3.2 Framework “História Técnica”**
Perfeito para posts narrativos.

**Estrutura:**
- O que aconteceu  
- O que parecia ser  
- O que descobri  
- O que aprendi  
- Como você pode aplicar  

---

## **3.3 Framework “Analogia → Conceito → Aplicação”**
Ótimo para explicar temas complexos.

**Estrutura:**
- Uma analogia simples  
- O conceito técnico  
- Um exemplo real  
- Uma dica prática  

---

## **3.4 Framework “Checklist”**
Bom para posts curtos e diretos.

**Estrutura:**
- Contexto rápido  
- Lista de pontos essenciais  
- Conclusão curta  

---

# 4. Modelos prontos de abertura

### **4.1 Pergunta provocativa**
- “Por que X acontece mesmo quando Y está certo?”  
- “Você já se perguntou por que seu banco faz Z?”  

### **4.2 Mini-história**
- “Ontem precisei resolver um problema que parecia simples…”  
- “Semana passada, um cliente me chamou por causa de…”  

### **4.3 Erro comum**
- “Se você ainda faz X, este post é para você.”  
- “O erro mais comum que vejo em ambientes Oracle é…”  

### **4.4 Curiosidade técnica**
- “Sabia que um único parâmetro pode causar…?”  
- “Pouca gente percebe, mas…”  

---

# 5. Modelos prontos de encerramento

### **5.1 Resumo + convite**
> “Em resumo, o ponto central é entender a causa antes do efeito.  
> Já passou por algo parecido? Vamos trocar ideias.”

### **5.2 Conclusão pessoal**
> “Esse caso reforçou para mim a importância de investigar o problema certo.”

### **5.3 Encerramento aberto**
> “Se quiser, posso mostrar no próximo post como diagnosticar isso em produção.”

---

# 6. Checklist final antes de publicar

### **6.1 Abertura**
- [ ] O gancho é forte?  
- [ ] A primeira frase prende atenção?  

### **6.2 Desenvolvimento**
- [ ] Expliquei o problema raiz?  
- [ ] Mostrei o raciocínio?  
- [ ] Dei exemplos práticos?  
- [ ] Usei analogias quando necessário?  
- [ ] Termos técnicos explicados na primeira vez?  

### **6.3 Linguagem**
- [ ] Tom leve e profissional  
- [ ] Frases curtas  
- [ ] Sem emojis (por enquanto)  

### **6.4 Encerramento**
- [ ] Resumo curto  
- [ ] Conclusão pessoal  
- [ ] Convite para interação
